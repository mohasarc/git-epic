# Review Rigor

Before writing any review verdict, run these gates and treat each as blocking the review (not optional). (1) **Root cause from source first** — state the failure mechanism (which value, which coordinate system, what changes on the triggering action) read from the code, not the diff; no mechanism, no verdict. (2) **Contribution test per hunk** — label every change `core | supporting | unrelated`; in a fix PR, `unrelated` is a finding by default, and recalibrated constants must show their arithmetic. (3) **Trace every value to its definition** with `file:line` — never reason about "the size/flag/state" abstractly. (4) **Negative space** — list what the PR claims to fix but doesn't touch, and what it touches that wasn't broken. (5) **Side-effect trace** — follow each new handler/listener to terminal effects (socket/network sends, server messages, persisted state) and flag any user-triggered side effect. (6) **Verify before labeling** — any "blocking / compile / lint / fails CI" claim requires the actual command output; mark each finding `observed` (ran it) or `reasoned` (inferred), and never assert confidence you didn't earn. (7) **Code shape fit** — check whether added or changed symbols, helpers, and file boundaries fit the local codebase’s stated conventions and nearby patterns; treat documented project conventions as review criteria, not optional style. (8) **Naming & shared shape** — scrutinize every new name and type shape against the project's documented conventions; don't pass them over just because they compile and read fine in isolation. Judge the diff against: right solution to the stated problem, minimal, changes nothing else, introduces no unwanted behavior, fully fixes it.
